# 成都英语地陪 Marketplace：MVP 讨论方案

**作者：Manus AI**  
**阶段：产品规划，尚未进入开发**

> **说明：** 我不是律师。下面有关公开资料导入的内容属于一般产品与风险管理建议，不构成法律意见；在正式批量采集、公开展示或商业化前，建议由熟悉中国个人信息保护、平台规则和旅游业务监管的律师审核。

## 一、建议的产品定位

第一阶段不应做成覆盖全国、支持即时付款的“大而全 OTA”，而应把产品定义为：

> **帮助来成都的外国游客，快速找到经过基本审核、能够用英语沟通的本地向导。**

成都单城切入是合理的，因为它能让供给审核、内容运营、客服和线下服务质量集中在一个市场完成。产品前台可以保留城市选择器，但 MVP 只开放 **Chengdu**，其余城市显示“Coming later”或暂不出现。

产品可参考 Showaround 的个人资料卡、价格、语言和兴趣标签，但中国市场版本需要增加三个差异化能力：**英语能力与身份分层、成都本地路线标签、外部资料的来源与本人认领机制**。

## 二、MVP 的目标用户

| 用户侧 | 首发人群 | 核心需求 | MVP 解决方式 |
|---|---|---|---|
| 游客端 | 第一次到成都的外国自由行游客、情侣、小家庭和三至六人小团 | 不熟悉本地交通、支付、景点预约和文化，希望有人用英语陪同 | 英语向导列表、结构化资料、路线标签、价格范围和询单表单 |
| 向导端 | 持证英语导游、具有稳定英语能力的本地陪同者、已经在小红书或抖音发布相关内容的人 | 低成本获得外国游客线索，并建立可信的英文资料页 | 自助申请、资料发布、管理员审核、询单通知和资料认领 |
| 平台运营端 | 成都本地运营人员 | 审核供给、处理举报、导入候选、跟踪询单和服务质量 | 管理后台、状态流转、来源记录和下架机制 |

供给端建议分层展示，而不是把所有人都统一称为“导游”。例如可以设置 **Licensed Guide、Local Host、Interpreter/Companion** 三类，并清楚标注其资质和服务边界。

## 三、游客端核心流程

游客进入首页后，先看到成都的视觉内容和一句清晰价值主张，然后通过日期、语言、服务类型、价格范围和兴趣标签筛选向导。列表卡片展示头像、英文名、英语水平、身份类型、服务价格、路线标签和响应状态。游客进入个人主页后，可以查看自我介绍、服务项目、可服务区域、可用时间、来源或认证状态，并提交询单。

MVP 建议采用 **Request to book / Send inquiry**，而不是即时付款。游客填写日期、人数、时长、期望路线和联系方式；向导接受后由双方进一步确认。这样可以先验证真实需求和供给响应，再决定是否引入付款、退款、取消政策和平台抽佣。

## 四、向导端核心流程

向导登录后完成一个分步骤申请表，包括个人介绍、英语能力、成都经验、服务类型、价格、服务区域、可用时间、头像及证明材料。提交后进入审核状态，通过后公开展示。

向导后台在 MVP 只需要四个模块：**资料编辑、服务与价格、可用状态、询单列表**。资质材料只用于审核，不应默认公开。平台应记录资料的审核状态、最后更新时间和向导是否已认领。

## 五、页面与信息架构

| 页面 | 主要内容 | MVP 必要性 |
|---|---|---|
| 首页 | 成都定位、搜索入口、精选服务类型、成为向导 CTA | 必须 |
| 向导列表 | 筛选、排序、资料卡和空状态 | 必须 |
| 向导详情 | 介绍、语言、价格、服务、区域、可用状态、询单入口 | 必须 |
| 提交询单 | 日期、人数、时长、路线偏好和联系方式 | 必须 |
| 成为向导 | 产品说明、申请资格和分步骤申请表 | 必须 |
| 向导工作台 | 资料、价格、可用状态和询单管理 | 必须 |
| 资料认领页 | 外部导入资料的身份验证、修改或下架申请 | 必须 |
| 管理后台 | 审核向导、审核导入资料、处理认领与举报 | 必须 |
| 支付与退款 | 在线付款、取消政策和结算 | 第二阶段 |
| 站内聊天 | 游客与向导实时消息 | 第二阶段；MVP 可先用询单与邮件通知 |
| 真实评价 | 仅允许完成真实服务后的用户评价 | 第二阶段；MVP 不展示虚构或预置评价 |

## 六、小红书与抖音资料导入方案

公开发布不等于可以无限制复制和商业化使用。即使个人信息已经公开，处理仍应限定在合理范围，并尊重个人明确拒绝或要求删除的权利。[1] 因此，MVP 不建议自动复制视频、正文、个人联系方式或整套主页，而应导入最小化的职业索引信息。

建议导入的字段包括：平台昵称、公开头像或占位图、服务城市、语言标签、简短事实性摘要、原帖链接、来源平台、导入日期和认领状态。手机号、微信号、邮箱、视频文件、完整正文以及用途不明的照片默认不导入。

| 方案 | 运行方式 | 权衡 | 成本 | 设置复杂度 |
|---|---|---|---|---|
| **人工策展导入** | 运营人员把公开帖子 URL 和最小职业信息录入后台，审核后上线；资料页突出“来源”和“认领此资料” | 速度较慢，但质量、隐私和来源最可控，适合成都冷启动 | 低，主要是运营时间 | 低 |
| **授权的自动导入** | 仅在获得平台许可、官方接口、授权数据供应商或发布者授权后，定期同步公开职业字段 | 扩张更快，但依赖授权、接口稳定性和持续合规审核 | 中到高 | 高 |

MVP 建议先采用第一种。后续如果候选量增长，再评估授权接口或合作数据源，而不是建设绕过登录、验证码或平台限制的爬虫。

导入资料的状态建议设计为：**Draft → Reviewed → Published as unclaimed → Claimed → Verified / Removed**。未认领资料必须标注来源和“尚未由本人确认”，并提供醒目的认领、修改、举报和下架入口。

## 七、商业模式的两种可行路径

| 路径 | 用户体验 | 优点 | 主要挑战 |
|---|---|---|---|
| **线索撮合模式** | 游客免费提交询单，平台向向导收会员费、优先展示费或合格线索费 | 上线快，暂时不处理支付和退款 | 必须控制垃圾询单，并证明线索质量 |
| **交易抽佣模式** | 游客在线预订并付款，平台按订单抽佣 | 收入与成交直接相关，交易闭环更完整 | 需要付款、退款、结算、争议处理及更严格的业务与合规设计 |

建议 MVP 先验证线索撮合，但数据模型应预留订单与支付状态，避免第二阶段重构。是否从第一天收费，可以独立决定；早期也可以先免费撮合，用真实转化率验证价值。

## 八、建议的数据模型

| 实体 | 关键字段 |
|---|---|
| User | 身份、角色、登录信息、状态 |
| GuideProfile | 英文名、简介、语言、向导类型、认证状态、城市、区域、响应状态 |
| Service | 服务名称、时长、价格单位、价格范围、路线标签 |
| Availability | 日期、可用状态、说明 |
| Inquiry | 游客、向导、日期、人数、需求、状态、来源渠道 |
| ImportedProfile | 来源平台、原帖 URL、公开昵称、导入摘要、认领状态、下架状态 |
| ClaimRequest | 认领人、验证材料、处理状态 |
| ModerationAction | 审核、拒绝、举报、修改和下架记录 |

## 九、四个阶段的产品路线图

| 阶段 | 目标 | 交付结果 |
|---|---|---|
| 1. 产品与视觉定稿 | 确定品牌、用户流程、资料字段和页面风格 | 可点击的信息架构与视觉方向 |
| 2. 核心 marketplace | 完成认证、向导申请、列表、详情与询单 | 可供真实向导和游客使用的 MVP |
| 3. 冷启动运营能力 | 完成候选导入、资料认领、审核和下架 | 可从公开线索建立初始成都供给库 |
| 4. 上线验证与优化 | 完成移动端、可访问性、测试和数据看板 | 可发布版本及首月运营指标 |

第一阶段暂不包含多城市、即时预订、在线支付、复杂日历同步、实时聊天和自动评价体系。这样可以避免在供需真实性尚未验证前投入过多。

## 十、建议的首发验证指标

这些数字是讨论用目标，不是行业基准。首发可以把“30–50 个审核通过的成都向导资料、每周 10 个合格询单、24 小时内回复率达到 70%、询单到双方建立联系的转化率达到 20%”作为早期验证门槛。只有在供给响应和询单质量稳定后，才扩大付费获客或开放第二个城市。

## 十一、现在需要共同决定的事项

| 决策 | 选项 A | 选项 B | 我的初步倾向 |
|---|---|---|---|
| 首发交易方式 | 只提交询单 | 即时预订与付款 | 先询单 |
| 供给范围 | 只收持证导游 | 持证导游与本地陪同者分层 | 分层展示，但身份清晰 |
| 外部资料 | 人工策展、原帖跳转、本人认领 | 自动抓取并公开完整资料 | 人工策展 |
| 联系方式 | 联系信息直接公开 | 通过平台表单转发 | 平台表单，降低骚扰 |
| 价格展示 | 精确价格 | From / price range | 价格范围，询单后确认 |
| 首发语言 | 英文界面 | 中英双语 | 游客端英文，向导后台可中英双语 |

只要以上六项达成一致，就可以进入 UI 设计和数据库结构确认。
